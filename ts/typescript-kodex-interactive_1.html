1Why TS Exists

The problem TS solves

What it is: TypeScript is a static type checker over JavaScript that flags type and shape mistakes before runtime.

Run this plain JavaScript. It has a serious bug. Watch what JS does about it — which is nothing.

● demo.js — plain JavaScript

NaN. No error, no warning — it runs and produces garbage. You'd find out in production, or worse, in front of a recruiter in a live coding round. TS moves this error from runtime (user sees it) to compile-time (you see it in your editor before you even save).

💡 Concept

TypeScript is not a new language. It's JavaScript + a checker that runs before your code runs, comparing the shapes of your values against the shapes you declared.

⚠ Pitfall

TS errors happen at compile time. At runtime, TS is gone — it compiles down to plain JS. That's also true of the ▶ Run buttons in these notes: they strip the types and execute, so runtime behavior shows here, while type errors are what the quizzes below are for.

🕹 Challenge 01 — Pick one
TypeScript makes your code run faster and more securely. True or false?
✅ Exactly. TS compiles away completely. Performance and security at runtime are untouched — it only changes when a whole class of bugs gets discovered: in your editor instead of in production.
❌ This is the #1 misconception. TS compiles down to plain JS — the browser/Node never sees your types. Zero runtime effect on speed or security. What changes is when you find bugs.
2Setup

Simple TypeScript project setup

What it is: the minimum local setup: TypeScript in the project, a tsconfig.json, and watch mode.

The current simplest setup is project-local TypeScript with a generated tsconfig.json. Start with npm, set the package type, then let the compiler drive watch mode.

💡 Concept

For a Node project, begin with npm init -y, install TypeScript locally, and set "type": "module" in package.json if you want modern ESM behavior. That keeps your project aligned with the current Node + TypeScript workflow.

setup-ts.sh
npm init -y
npm install -D typescript @types/node
# package.json
"type": "module"
npx tsc --init

# watch type-checking and keep the output visible
npx tsc --watch --preserveWatchOutput
✅ What to run while coding

Use tsc --watch for continuous type-checking, and add --preserveWatchOutput so the log stays on screen instead of clearing on every rebuild.

⚠ Pitfall

If you use "type": "module", keep your runtime and compiler settings in sync. Modern Node projects usually pair it with a Node-friendly module setting in tsconfig.json rather than mixing module systems casually.

3Core Types

Primitives & type inference

What it is: primitives are base value categories; inference is TS auto-detecting types from assigned values.

💡 Concept

Primitive data types are the simplest values TypeScript works with: string, number, boolean, null, undefined, bigint, and symbol. They are not objects and usually hold one single value.

Type inference means TypeScript decides a variable's type for you from the value you assign, so you do not always need to write the type yourself.

primitives.ts
let age: number = 25;      // explicit annotation
let city = "Bhopal";         // inferred as string — no annotation needed
let isLive: boolean = true;

Rule of thumb: if TS can figure it out from the value, don't annotate. Annotate when there's no initial value, or the shape isn't obvious (function params, empty arrays, API responses).

⚠ Pitfall

Over-annotating everything (let name: string = "Ankur" on every line) isn't wrong — just noisy, and in a code review it signals "doesn't trust inference."

🌀 Common Confusion

let x: number (annotation — a promise about future values) vs let x = 5 (inference — TS reading the current value). Both end up type-checked identically; only one requires writing the type.

🕹 Challenge 02 — Which one needs an annotation?
One of these genuinely needs a type annotation. Which?
✅ Right. With no initial value, TS infers any for result — the checker basically gives up on it. Option A has a value, so inference handles it perfectly.
❌ Option A already has a value (499) — TS infers number on its own. Option B is the trap: declared with no value, TS silently infers any, and type-checking is gone for that variable.
4Core Types

Arrays & tuples

What it is: arrays are variable-length lists of one item type; tuples are positional lists with fixed index types.

arrays.ts
let scores: number[] = [10, 20, 30];
scores.push(40); // normal array growth

// tuple: fixed positions + known type per position
let point: [number, number] = [28.6, 77.2];
point.push(99); // allowed on mutable tuples

// use readonly tuple when you want to block push/pop
const lockedPoint: readonly [number, number] = [28.6, 77.2];
💡 Concept

Array = "any number of items, all one shape." Tuple = "positions have meaning, and each position has a known type." Tuples earn their place where order matters — like useState()'s [value, setValue].

⚠ Pitfall

let point = [28.6, 77.2] without annotation is inferred as number[] — a plain array, not a tuple. Also, mutable tuples can still allow push; if you want a strict "no push/pop" tuple, use readonly [number, number].

🕹 Challenge 03 — Will this compile?
let user: [string, number] = [25, "Ankur"];
✅ Correct. A tuple isn't a bag of allowed types — each position has a fixed type. Position 0 must be string, position 1 must be number. Order is the whole point of a tuple.
❌ Tempting — both a string and a number are there. But a tuple locks types per position: index 0 must be string, index 1 must be number. Swapped = two errors.
5Core Types

Objects: interface vs type

What it is: both define shapes; interface is object-contract syntax, while type is general alias syntax.

shapes.ts
interface User {
  name: string;
  age: number;
  isAdmin?: boolean;  // optional — the ? is doing real work
}
💡 Concept

Practical rule for real MERN codebases: interface for object shapes, type for everything else (unions, function signatures, tuples).

Why interface exists: it models contracts between parts of your app. A controller can promise "I accept a UserInput", a service can promise "I return a UserResponse." This keeps large teams aligned because everyone codes against the same named shape.

Why teams prefer it for objects: interfaces are easy to extend and can be merged across files (useful for framework types and plugin ecosystems). So object models can grow over time without rewriting every type from scratch.

why-interface.ts
// 1) EXTEND existing object contracts cleanly
interface Person {
  name: string;
}

interface Admin extends Person {
  role: "admin";
}

// 2) DECLARATION MERGING (same name, combined shape)
interface RequestMeta {
  requestId: string;
}
interface RequestMeta {
  ip: string;
}

const meta: RequestMeta = { requestId: "abc", ip: "127.0.0.1" };
✅ Decision Rule (simple)

Use interface when describing object contracts that may evolve. Use type when composing non-object forms like unions ("open" | "closed"), tuples, mapped/conditional aliases, or function signatures.

🌀 Common Confusion

{ age: number } looks like a JS object literal { age: 25 } but the colon means something different — it's a type annotation, not an assignment. Students coming straight from JS misread "age equals the word number."

🕹 Challenge 04 — Fix the error
This errors: "Property 'isAdmin' is missing." Which fix is the right instinct? 
interface User { name: string; isAdmin: boolean; }
const u: User = { name: "Riya" };  // ❌
✅ Right instinct. If a field genuinely may be absent, say so in the type with ? — now the compiler protects every place that reads u.isAdmin, forcing an undefined-check.
❌ any makes the error disappear — along with all checking on u, forever. You didn't fix the bug, you fired the security guard. The honest fix: mark the field optional with ?.
6Core Types

Union types & literal types

What it is: unions allow one of multiple types; literal types allow only exact values.

unions.ts
let id: string | number;   // either is fine

// literal type — not "a string", but THIS exact string
let status: "pending" | "success" | "failed";
💡 Concept

This is where TS starts feeling powerful, not just strict. A literal union turns a typo into a compile error instead of a silent bug where your if/else chain just never matches.

⚠ Pitfall

Lowercase string is a category (any string). Quoted "pending" is a single member of that category treated as its own type. Mixing these up makes literal unions read like nonsense.

🕹 Challenge 05 — Which line errors?
Given let status: "pending" | "success" | "failed"; — which assignment fails?
✅ Sharp eye — capital P. Literal types are exact-match, case-sensitive. "Pending" ≠ "pending". This is exactly the class of bug that silently breaks if/else chains in plain JS — TS catches it before you run anything.
❌ "failed" is in the allowed set — that one's fine. Look closer at Option B: capital P. Literal types are exact, case-sensitive matches, and "Pending" isn't in the union.
7Core Types

Type narrowing

What it is: narrowing is TS reducing a broad union into a specific type after runtime checks.

Narrowing is TS watching your if-checks and shrinking a union to one specific type inside each branch. Run this and check the output:

● narrowing.ts — editable, try changing the argument
🌀 Common Confusion

typeof id === "string" is plain JS that already existed — narrowing adds nothing to the compiled output. TS is just smart enough to read that check and adjust types inside the branch.

🕹 Challenge 06 — Will this compile?
function printId(id: string | number) {
  console.log(id.toUpperCase());  // no typeof check
}
✅ Correct. TS refuses to gamble. On a string | number, you can only use what both types support — until a guard proves which one you have. That's the entire deal with unions.
❌ "It could be a string!" — that's exactly the reasoning TS rejects. It could also be a number, and (42).toUpperCase() would crash. No benefit of the doubt: narrow first, then use.
8Functions

Function parameter & return typing

What it is: function typing is a compile-time contract for inputs (parameters) and output (return type).

Remember the total("499", undefined) bug from Topic 1? Same function, now in TypeScript. The types are stripped when you hit Run — but in your editor, that call would be a red squiggle before you even save.

● functions.ts — fix the call so the output is correct
🌀 Common Confusion

Arrow function return types go after the params, before the arrow: const total = (price: number, qty: number): number => price * qty. Students often try to place it after the arrow.

🕹 Challenge 07 — Why annotate the return type?
TS can infer return types automatically. So what does writing : number explicitly actually buy you?
✅ Exactly. Declare : number and accidentally return "done" in some edge-case branch — TS flags it at the return statement. Without the annotation, TS just widens the inferred type to number | string and the bug travels downstream.
❌ It reads like documentation, but it's enforced. If any branch returns a string when you declared : number, the error lands exactly on that return. Without it, TS silently infers number | string and the problem surfaces somewhere far away.
9Functions

Optional & default parameters

What it is: optional params may be omitted; default params auto-fill missing values.

optional-params.ts
function greet(name: string, greeting?: string) {
  console.log(`${greeting ?? "Hello"}, ${name}`);
}

function greet2(name: string, greeting: string = "Hello") {
  console.log(`${greeting}, ${name}`);
}
💡 Concept

greeting? → inside the function, type is string | undefined, you handle the fallback. Default value = "Hello" → inside the function it's just string, because TS knows the fallback already ran.

🕹 Challenge 08 — Which signature is legal?
One of these is a straight-up syntax error. Which one works?
✅ Optional params must come after required ones. Think about the call site: f(5) — if the optional came first, which parameter did 5 fill? Ambiguous, so TS bans it.
❌ Option A is the error: "a required parameter cannot follow an optional parameter." Reason: at the call site f(5), TS couldn't tell whether 5 is a or b. Optionals always go last.
10Functions

any vs unknown

What it is: both can hold any value, but any disables checks while unknown requires proof before use.

The single most important topic in this class. Both can hold anything — the difference is what happens next.

any-vs-unknown.ts
let a: any = JSON.parse("{}");
a.anything();   // ✅ compiles — then explodes at runtime

let u: unknown = JSON.parse("{}");
u.anything();   // ❌ compile error — prove what it is first
⚠ Pitfall

any is a type-safety black hole — it silently disables checking on everything derived from it. One any from a bad API response can erase type safety across an entire file. If you're tempted to write any to make an error go away, reach for unknown and ask why the compiler is confused.

🕹 Challenge 09 — Translate to plain language
What does any actually say to the compiler?
✅ Brutal but accurate. any isn't a flexible wildcard — it's an off-switch, and it spreads: everything derived from an any becomes any too. The "flexible wildcard that's still checked" you were imagining? That's unknown.
❌ That's what everyone assumes — and that description actually fits unknown, which stays safe by forcing a check before use. any is the opposite: checking fully off, and it spreads to everything derived from it.
11Working at Scale

enum vs union of literals

What it is: enum creates a runtime object; a literal union is a compile-time-only set of allowed values.

enum-vs-union.ts
enum Role { Admin = "ADMIN", User = "USER" }

// the modern preference:
type Role2 = "ADMIN" | "USER";
💡 Concept

Simple rule: both restrict values to a fixed set, but they behave differently.

enum exists at runtime as an object. A literal union exists only during type-checking.

Use literal unions by default in app code. Use enum when you need a shared named constant object at runtime.

✅ What is enum?

An enum is a named set of constants. TypeScript also generates a JavaScript object for it.

Example: enum Role { Admin = "ADMIN", User = "USER" } lets you write Role.Admin in code and also use Role at runtime.

Equivalent union type check: type Role = "ADMIN" | "USER" (no runtime object generated).

🕹 Challenge 10 — Predict, then run below to verify
You compile a file containing an enum. Does the enum appear in the output JavaScript?
✅ Correct — and this makes enum weird. Almost every TS construct vanishes on compile, but enum generates a real lookup object in your bundle. Verify it yourself in the runner below.
❌ Great rule, wrong exception. "Types are erased" is true for interfaces, unions, annotations — but enum breaks it: it compiles to a real JS object that ships in your bundle. See for yourself below.

12Working at Scale

keyof & typeof

What it is: keyof T gives a union of property names of T; type-position typeof v gives the type shape of value v.

Goal: just enough to read real codebases confidently. Not a generics deep-dive.

keyof-typeof.ts
interface User { name: string; age: number; }

type UserKeys = keyof User;      // ?

const config = { port: 3000, env: "dev" };
type ConfigType = typeof config;  // { port: number; env: string }
⚠ Pitfall

Type-position typeof and Topic 6's runtime typeof are the same keyword doing two completely different jobs depending on whether it appears inside a type annotation or in normal code. Point at both examples side by side when someone looks lost.

🕹 Challenge 11 — What is UserKeys?
Given the interface above, what does keyof User produce?
✅ keyof extracts the key names as a literal union — not the value types. Option B would be the value types, which is a different tool. Payoff: when the interface changes, the derived union updates automatically instead of going stale.
❌ You grabbed the value types. keyof takes the key names: "name" | "age" as string literals. Notice it connects two things you already learned — keys extracted into a literal union (Topic 5).
13Working at Scale

Classes & access modifiers

What it is: TS classes are JS classes + typed fields, access modifiers (public/private/protected/readonly), and constructor shorthand.

classes.ts
class UserService {
  // long form: declare fields, assign in constructor
  private db: Database;
  readonly serviceName = "users";

  constructor(db: Database) {
    this.db = db;
  }
}

// constructor SHORTHAND — modifier on the param creates + assigns the field
class UserService2 {
  constructor(private db: Database, readonly serviceName = "users") {}
}

// classes can IMPLEMENT an interface — the contract from Topic 5
interface Repo {
  findById(id: string): Promise<unknown>;
}
class UserRepo implements Repo {
  async findById(id: string) { return {} }
}
💡 Concept

Two things TS adds over JS classes: access modifiers (who can touch a field) and constructor parameter shorthand (writing private db: Database in the constructor declares the field AND assigns it — one line instead of three). You'll see the shorthand everywhere in NestJS-style dependency injection.

Class vs interface: an interface is a type-only contract (erased at compile time). A class is both a type and a runtime value — it produces real JS. That's why a class can implements an interface: the interface says what, the class provides how.

⚠ Pitfall

private is compile-time only. It stops your TypeScript code from accessing the field — but after compilation the field is a completely normal, fully accessible JS property. If you need true runtime privacy, that's JavaScript's own #field syntax, which actually enforces it in the engine. Demo this in the runner below — it surprises everyone.

🌀 Common Confusion

"If I have interfaces, why do I need classes?" — Interfaces describe shape; classes bundle shape + behavior + construction. Use an interface when you just need a contract (API payloads, function params). Reach for a class when you need instances with methods, shared logic, or dependency injection. In typical MERN route/controller code, you'll write far more interfaces than classes — that's normal.

🕹 Challenge 12 — Does private survive?
A field is marked private apiKey: string. After the TS is compiled to JS and running in Node — can other code read obj.apiKey?
✅ Correct — and this matters for security thinking. private is a message to the compiler and your teammates, not to the runtime. Never rely on it to protect secrets. True runtime privacy is JS's #field syntax. Same "types are erased" rule from Topic 1, applied to modifiers.
❌ Feels like it should, right? But private is erased at compile time along with every other type annotation — at runtime it's a plain property anyone can read. If you need the engine to actually enforce privacy, that's JS's native #field. Run the proof below.

14Working at Scale

Utility types: Partial, Pick, Omit

What it is: built-in type transformers that derive new types from existing ones — instead of writing near-duplicate shapes by hand.

This is keyof's payoff. You already have a User type — utility types let you carve new shapes out of it that stay in sync automatically.

utility-types.ts
interface User {
  id: string;
  name: string;
  email: string;
  password: string;
}

// Partial — every field becomes optional. Perfect for PATCH updates:
type UserUpdate = Partial<User>;
// { id?: string; name?: string; email?: string; password?: string }

// Pick — keep ONLY these keys:
type UserPreview = Pick<User, "id" | "name">;
// { id: string; name: string }

// Omit — keep everything EXCEPT these keys:
type SafeUser = Omit<User, "password">;
// { id: string; name: string; email: string }
💡 Concept

Rule of thumb: Pick when you want a few fields, Omit when you want to remove a few fields, Partial for update payloads. The win over hand-writing the shape: add a field to User tomorrow, and SafeUser updates itself — a hand-written duplicate silently goes stale. Same source-of-truth idea as keyof/typeof, one level up.

This connects directly to the next block: your API's response type can be derived — type UserResponse = Omit<UserDoc, "hashedPassword" | "_id"> & { id: string } — instead of written by hand and hoped-kept-in-sync.

⚠ Pitfall

Omit doesn't error on keys that don't exist. Omit<User, "pasword"> (typo!) compiles fine and silently omits nothing — the password stays in the type. Pick is stricter and rejects unknown keys. If you want typo-safe omission, that's exactly why some teams write Omit<User, keyof Pick<User, "password">> patterns — but for this class, just know: double-check your Omit keys.

🌀 Common Confusion

Partial<User> makes fields optional in the type — it doesn't fill in defaults or merge objects at runtime. Students write a PATCH handler with Partial<User> and expect the missing fields to "keep old values" automatically. That merge is your job ({ ...existing, ...update }) — Partial just makes the compiler allow the missing fields.

🕹 Challenge 13 — Pick the right tool
You're building the response type for GET /users/:id. It should be the full User minus password. Which derives it best?
✅ Right tool. Omit removes exactly the dangerous field and keeps the rest required — the response contract stays strong. And when User grows a new safe field next sprint, Res picks it up automatically.
❌ Partial makes everything optional — including fields your response always sends. Worse: password is still in the type, just optional, so nothing stops it leaking. You wanted removal, not optionality: that's Omit<User, "password">.
15Payoff: Express

Typing req.body, req.params, req.query

What it is: adding explicit types to request params, body, and query so handlers know exact input shapes.

Everything so far was setup for this block. Here's the moment TS meets your actual Express code.

route.ts
import type { Request, Response } from "express";

interface CreateUserBody {
  name: string;
  email: string;
}

// Request<Params, ResBody, ReqBody, ReqQuery> — positional!
app.post("/users", (req: Request<{}, {}, CreateUserBody>, res: Response) => {
  const { name, email } = req.body; // fully typed, autocomplete works
});
⚠ Pitfall

The generic slots are positional and easy to get backwards — it's Request<Params, ResBody, ReqBody, ReqQuery>. Skip the first two {} placeholders and you've typed req.params as your body shape.

🕹 Challenge 14 — Spot the mistake
A student writes Request<CreateUserBody> and complains that req.body is still any. What actually happened?
✅ Exactly. Slot 1 = Params, slot 2 = ResBody, slot 3 = ReqBody. Their body type landed in the params slot, and body stayed any. Fix: Request<{}, {}, CreateUserBody>.
❌ Reinstalling node_modules is the classic panic move — but nothing's broken. The generic is positional: slot 1 is Params, and their body type went there. Body needs slot 3: Request<{}, {}, CreateUserBody>.
🌀 Common Confusion — say this out loud in class

Typing req.body does not validate anything at runtime. It's a promise to the compiler, not a guard against a malicious request — TS trusts your annotation even if the client sends garbage. Runtime validation (Zod, Joi, manual checks) is a separate, still-necessary step. Nobody should walk out thinking TS = input validation.

16Payoff: Express

Request / Response types from Express

What it is: using Express-provided TypeScript types to annotate middleware and route handlers consistently.

imports.ts
// idiomatic type-only import — zero runtime presence, fully erased
import type { Request, Response, NextFunction } from "express";
⚠ Pitfall

Without @types/express installed, these types fail to resolve or silently fall back to any. Students think "TS isn't working" — really the type definitions were never installed. First debugging step whenever Express types feel dead: check package.json.

🕹 Challenge 15 — Why the separate package?
Why does Express need @types/express at all, when packages like Mongoose don't need a separate types package?
✅ Right. Express predates TS's dominance and ships no types of its own. The community (DefinitelyTyped) maintains @types/express. Libraries written in TS — like modern Mongoose — bundle their own types, so no extra install.
❌ Nothing paid — it's free and community-maintained (the DefinitelyTyped project). The real reason: Express is written in plain JS with no built-in types, so someone had to describe its shapes to TS. TS-native libraries bundle their own.
17Payoff: Express

One shape isn't enough: Input vs DB vs Response

What it is: separate type models for incoming payloads, stored database documents, and outgoing API responses.

The highest-value pattern in the session — it maps directly to a bug every student will ship: trusting req.body blindly and leaking DB fields in responses.

user.types.ts
// what the CLIENT sends — no id, no timestamps, raw password
interface UserInput {
  name: string;
  email: string;
  password: string;
}

// what MongoDB stores — _id, hashedPassword, timestamps
interface UserDoc {
  _id: Types.ObjectId;
  name: string;
  email: string;
  hashedPassword: string;
  createdAt: Date;
}

// what you SEND BACK — never the password. ever.
interface UserResponse {
  id: string;
  name: string;
  email: string;
}
💡 Concept

A "user" isn't one type — it's a different shape at every stage of its life. Input (raw, untrusted) → DB doc (full, internal fields) → response (trimmed, safe). Naming these separately forces a conscious decision about what crosses each boundary, instead of leaking hashedPassword because one loose type got reused everywhere.

✅ Now that you know Omit

You don't have to hand-write UserResponse anymore. Derive it from the source of truth: type UserResponse = Omit<UserDoc, "hashedPassword" | "_id" | "createdAt"> & { id: string }. Add a safe field to UserDoc tomorrow, and the response type updates itself — while hashedPassword stays permanently excluded.

⚠ Pitfall

Real bug from building this pattern: importing a bare ObjectId from mongoose vs Types.ObjectId — two different classes with the same name from different namespaces, producing a confusing structural mismatch error. Always Types.ObjectId for Mongoose documents.

🕹 Challenge 16 — Catch the security bug
A teammate writes this login handler. What's wrong? 
const user: UserDoc = await UserModel.findOne({ email });
res.json(user);
✅ This is the whole point of three shapes. UserDoc includes hashedPassword — sending it raw exposes hashes to every client (and every logging proxy in between). The fix: map to a UserResponse before res.json(). Trusted source ≠ safe to expose.
❌ "Trusted database" is exactly the trap — the question isn't whether the data is trustworthy, it's whether every field should reach the client. hashedPassword just got shipped to the browser. Map to UserResponse first.
18Finale

Fix the error — rapid fire

What it is: short drills to practice reading TS errors and choosing root-cause fixes.

Three real TS errors. For each: read the error, pick the right fix. This is the exact skill that makes TS errors stop being scary.

🕹 Finale 1 of 3
"Argument of type 'string' is not assignable to parameter of type 'number'." 
total("499", 2);
✅ You fixed the actual problem — the value really was the wrong type, so convert it. as any would have made the error disappear while keeping the bug: "499" * 2 happens to work via coercion, until one day it doesn't.
❌ as any silences the compiler but the string is still a string — you kept the bug and deleted the warning. When TS says types disagree, the fix is almost always at the data, not at the annotation.
🕹 Finale 2 of 3
"Property 'toUpperCase' does not exist on type 'string | number'." 
function show(id: string | number) {
  return id.toUpperCase();
}
✅ The error was TS protecting you — id might be a number, and (42).toUpperCase() crashes at runtime. Narrowing handles both cases honestly. any would compile fine and crash in production the first time a number arrives.
❌ With any it compiles — and then crashes at runtime the first time someone passes a number. The error message told you exactly what's missing: proof that id is a string. That proof is a typeof narrow.
🕹 Finale 3 of 3
"Property 'isAdmin' is missing in type '{ name: string; }' but required in type 'User'." The API you're consuming genuinely doesn't always send isAdmin. Best fix?
✅ Key principle to end on: types should describe reality, not force reality to fake it. If the field genuinely may be absent, the type says so — and now every consumer is forced to handle the missing case. Hardcoding false at call sites invents data and hides the truth.
❌ Hardcoding isAdmin: false everywhere invents data that doesn't exist — and if 5 call sites do it, one of them will eventually invent true. The honest fix: the type should describe reality (?), and let the compiler force consumers to handle absence.
✅ Closing note for students

Every error you just fixed follows one shape: TS compares the type you promised against the type you're actually using, and tells you exactly where they disagree. Once you can read that sentence, you can read almost any TS error — including the scary ones with generics in them. And notice: in every single challenge, as any was on the ballot and it was never the answer.
