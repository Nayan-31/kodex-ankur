Backend · MERN · Module 01

Docker, explained the way you'll actually use it.

Your Node + Express + MongoDB backend runs perfectly on your laptop, then dies on your friend's machine and again on the server. This is the problem Docker was built to kill. We'll start with the pain, then build up the fix — piece by piece.

node 20 express mongodb docker docker compose
01
The Problem — "Works on my machine"
Before the solution, feel the pain. This is why Docker exists.

You build a MERN backend. Node 20, Express, a few npm packages, and it connects to MongoDB. On your laptop, npm start works flawlessly. Then you do one of these things:

  • You send the project to a teammate to run.
  • You deploy it to a server (EC2, a VPS, anywhere).
  • You come back to the project six months later on a fresh OS install.

And suddenly — it breaks. The same code. The errors look like this:

your teammate's terminal
# they have Node 16, you built on Node 20
SyntaxError: Unexpected token '?' # optional chaining not supported

# or a native package compiled for your OS, not theirs
Error: Cannot find module 'bcrypt'
# ...node-gyp rebuild failed

# or Mongo isn't even installed on the server
MongooseServerSelectionError: connect ECONNREFUSED 127.0.0.1:27017

Your code is fine. What differs is everything around the code — the Node version, the OS, the installed system libraries, whether Mongo exists, which environment variables are set. This invisible bundle is called the environment, and it never travels with your code.

The Core Problem

Code is only half of what runs. The other half is the environment — OS, runtime version, system libraries, dependencies, config. You ship the code but not the environment, so the same code behaves differently everywhere.

How we used to "solve" it (and why it hurts)

Old approachWhat goes wrong
A README with setup steps "Install Node 20, then Mongo, then set these 8 env vars..." — humans skip steps, versions drift, it rots.
A full Virtual Machine Ships a whole OS per app. 5–10 GB each, slow to boot, heavy on RAM. One VM per service is painful.
"It works, don't touch the server" The server becomes a fragile pet nobody can rebuild. Lose it and you lose everything.

We need a way to package the code together with its entire environment, so the exact same thing runs identically on any machine — laptop, teammate's PC, or production server. That package is what Docker gives us.

02
What Docker Is
The fix: package code + environment into one portable unit.

Docker lets you bundle your app and everything it needs to run into a single, sealed package called a container. The container carries the OS layer, Node, your code, your dependencies — all of it. Ship that container anywhere Docker is installed, and it runs the same. The "works on my machine" excuse dies, because now your machine is the container, and you ship the whole machine.

One-line definition

Docker packages your application and its complete environment into a container — an isolated, portable unit that runs identically on any machine with Docker installed.

Container vs Virtual Machine — why containers won

This is the question every beginner asks: "isn't this just a VM?" No — and the difference is exactly why Docker is fast and light.

WhatVirtual MachineDocker Container
What it ships A full guest OS + your app Just your app + its dependencies ✓
Size Several GB each Tens to hundreds of MB ✓
Boot time Minutes Seconds or less ✓
OS kernel Each VM runs its own kernel All containers share the host kernel ✓
Run 5 services? 5 heavy VMs, lots of RAM 5 light containers, minimal overhead ✓
Why containers are light

A VM virtualizes hardware and boots a whole OS on top. A container virtualizes only the operating system — it shares the host's kernel and isolates just the process. Less to ship, less to boot, less to run.

💡

Containers feel like lightweight VMs, but they are really just isolated processes on your machine that think they have their own filesystem, network, and dependencies.

03
The Mental Model
Three words that unlock everything: Dockerfile, Image, Container.

Almost all Docker confusion comes from mixing up three words. Lock these down now and the rest is easy. Use the cooking analogy — it always works:

TermCooking analogyWhat it actually is
Dockerfile the recipe A text file of instructions: which base to use, what to copy, what to install, how to start.
Image the frozen meal The built result of the recipe. A read-only, shippable snapshot. Build once, reuse many times.
Container the served dish A running instance of an image. You can run many containers from one image at once.

The flow is always one direction — you write a recipe, build it into an image, then run that image as a container:

Write a Dockerfile
You author the instructions. "Start from Node 20, copy my code, install deps, run npm start."
docker build → Image
Docker reads the recipe and bakes a frozen, read-only snapshot. This is what you share or push.
docker run → Container
Docker spins up a live, running instance of the image. This is your actual running backend.
💡

One image, many containers. Build your API image once, then run three copies behind a load balancer — same image, three containers. This is the foundation of scaling.

Where do images come from? — Docker Hub

You rarely start from nothing. Docker Hub is a public registry of pre-built images. node, mongo, redis, nginx — all official images you can pull and build on top of. Your Dockerfile's first line is almost always "start from one of these."

04
Images & Containers, Hands-On
Pull a real image and run it before we build our own.

Best way to understand the image → container jump is to do it once with something that already exists. Let's pull the official Node image and poke at it.

Step 1 — Pull an image from Docker Hub
Terminal 
docker pull node:20-alpine

This downloads the Node 20 image (the alpine tag is a tiny Linux base — more on that later). It now lives on your machine, frozen and ready.

Verify — list your images
Terminal 
docker images
Step 2 — Run it as a container
Terminal 
docker run -it node:20-alpine node -v
# prints v20.x.x — Node running inside an isolated container,
# even if you have NO Node installed on your host machine
Let this sink in

You just ran Node 20 without installing Node. The container brought its own. This is the whole point — the environment travels inside the image.

Verify — see running & stopped containers
Terminal 
docker ps          # currently running containers
docker ps -a       # all containers, including stopped ones
05
The Dockerfile
The recipe for your own backend image, line by line.

A Dockerfile is a plain text file named exactly Dockerfile (no extension) at the root of your project. Each line is an instruction Docker runs top to bottom to build your image. Here's a complete one for an Express API — read it first, then we'll break down every line.

Dockerfile
FROM node:20-alpine

WORKDIR /app

COPY package*.json ./

RUN npm install

COPY . .

EXPOSE 5000

CMD ["node", "server.js"]
InstructionWhat it doesWhy it matters
FROM Sets the base image to build on top of. You inherit a working Node 20 + Linux. Never start from scratch.
WORKDIR Sets the working directory inside the container. All later commands run from /app. Keeps things tidy.
COPY package*.json Copies only the package files first. Layer caching — deps only reinstall when package.json changes, not on every code edit.
RUN Executes a command at build time. Installs your dependencies into the image, so they ship with it.
COPY . . Copies the rest of your source code in. After deps are cached — your actual app code.
EXPOSE Documents which port the app listens on. A signpost for humans & tools. Doesn't publish the port by itself.
CMD The command run when the container starts. RUN is build-time; CMD is run-time. This launches your server.
RUN vs CMD — the classic confusion

RUN happens once, while building the image (e.g. installing packages). CMD happens every time you start a container from that image (e.g. launching your server). Build-time vs run-time.

The .dockerignore file — don't skip this

Just like .gitignore, a .dockerignore keeps junk out of your image. Most important: never copy node_modules from your host — they may be built for your OS and will conflict with the npm install done inside the container.

.dockerignore
node_modules
npm-debug.log
.env
.git
Dockerfile
.dockerignore
⚠️

Never copy host node_modules into an image. Native packages like bcrypt are compiled for a specific OS. The container installs its own — let it. Copying yours causes the exact "Cannot find module" errors we saw in Section 01.

Core Commands You'll Use Daily
The 10 commands that cover 90% of real work.
Build an image from your Dockerfile
Terminal 
docker build -t my-api .
FlagValueWhat it does
-tmy-apiTags (names) the image so you can refer to it later.
.(current dir)The build context — where the Dockerfile and code live.
Run a container from the image
Terminal 
docker run -d -p 5000:5000 --name api my-api
FlagValueWhat it does
-d—Detached — runs in the background, frees your terminal.
-p5000:5000Maps host port → container port (covered next section).
--nameapiGives the container a friendly name instead of a random one.
Inspect & debug
Terminal 
docker ps                      # what's running
docker logs -f api            # stream logs (-f = follow)
docker exec -it api sh         # open a shell INSIDE the running container
💡

docker exec -it api sh is your best friend for debugging — it drops you inside the live container so you can check files, env vars, and network from the app's point of view.

Stop & clean up
Terminal 
docker stop api              # stop a running container
docker rm api                # remove a stopped container
docker rmi my-api            # remove an image
docker system prune -a        # nuke unused images/containers (reclaim disk)
⚠️

docker system prune -a deletes all stopped containers and unused images, not just dangling ones. Great for reclaiming gigabytes — but be sure you don't need them first.

07
Dockerize a Real Express API
End to end — from source code to a running container.

Let's take a minimal Express backend and containerize it fully. Here's the app we're shipping:

server.js
const express = require("express");
const app = express();

const PORT = process.env.PORT || 5000;

app.get("/", (req, res) => {
  res.json({ status: "API is running inside Docker 🐳" });
});

app.listen(PORT, () => {
  console.log(`Server on port ${PORT}`);
});
💡

Notice process.env.PORT || 5000. Reading config from the environment (not hardcoding it) is what makes the same image reusable across dev, staging, and prod. We'll feed that env var in the next section.

Project structure

my-api/
my-api/
├── server.js
├── package.json
├── Dockerfile
└── .dockerignore
Step 1 — Build the image
Terminal 
docker build -t my-api .
Step 2 — Run the container
Terminal 
docker run -d -p 5000:5000 --name api my-api
Step 3 — Verify it works
Terminal 
curl http://localhost:5000
# {"status":"API is running inside Docker 🐳"}
What just happened

Your laptop has no idea what Node version the app needs — and doesn't care. The container brought Node 20, installed its own dependencies, and served the API. Hand this image to anyone with Docker and they get the identical result. The Section 01 problem is solved.

08
Ports & Environment Variables
Two things every backend needs, and where beginners get stuck.

Port mapping — the container is an island

A container is isolated by default. Your app listening on port 5000 inside the container is invisible from your laptop until you build a bridge. That bridge is -p host:container.

Your browser → localhost:3000
    ↓ -p 3000:5000
Container port 5000
    ↓
Express app responds ✅
Reading -p left to right

-p 3000:5000 means "traffic hitting host port 3000 forwards to container port 5000." The left number is the one you type in your browser; the right is what the app listens on inside.

⚠️

Forgot -p entirely? The container runs fine, logs look healthy — but localhost:5000 refuses to connect. No bridge, no access. This trips up nearly every beginner once.

Environment variables — config without rebuilding

Secrets and config (DB URLs, ports, API keys) must never be baked into the image. Inject them at run time so one image works everywhere.

Pass vars inline with -e
Terminal 
docker run -d -p 5000:5000 \
  -e PORT=5000 \
  -e MONGO_URI=mongodb://host.docker.internal:27017/mydb \
  --name api my-api
Cleaner — load from an env file
Terminal 
docker run -d -p 5000:5000 --env-file .env --name api my-api
💡

Same image, different env file per environment — dev, staging, production. Build once, configure at run time. This is the whole reason we read process.env.PORT instead of hardcoding.

host.docker.internal

Inside a container, localhost means the container itself, not your laptop. To reach a service running on your host machine (like a local Mongo), use host.docker.internal instead of localhost.

09
Volumes — Don't Lose Your Data
Containers are disposable. Your database can't be.

Here's a brutal lesson best learned early: containers are temporary. Anything written inside a container's own filesystem vanishes the moment it's removed. If your MongoDB stores its data inside the container and you run docker rm — every record is gone.

Mongo container running, data inside it
    ↓ docker rm mongo
All data destroyed ❌

Mongo container + volume
    ↓ docker rm mongo
Data survives in the volume ✅
Key Concept

A volume is storage that lives outside the container's lifecycle, managed by Docker on the host. Containers come and go; the volume — and your data — persists.

Two kinds of mounts you'll use

TypeSyntaxUse it for
Named volume -v mongo-data:/data/db Persisting database data. Docker manages where it lives on disk.
Bind mount -v $(pwd):/app Dev hot-reload — maps your live source folder into the container.
Run MongoDB with a persistent volume
Terminal 
docker run -d -p 27017:27017 \
  -v mongo-data:/data/db \
  --name mongo mongo:7

Now docker rm mongo and re-run it — your data is still there, restored from the mongo-data volume.

💡

Bind mounts for development are a superpower: mount your code into the container so edits on your laptop reflect instantly inside it — no rebuild needed. Pair with nodemon for live reload.

10
Docker Compose — API + Database Together
Real backends are never one container. Orchestrate them with one file.

Your MERN backend isn't just Node — it's Node and MongoDB, maybe Redis too. Running each by hand with long docker run commands, in the right order, wiring up networks — that's misery. Docker Compose defines all of it in one YAML file and starts everything with a single command.

Why Compose exists

One docker-compose.yml describes every service, their config, volumes, and how they connect. docker compose up brings the whole stack online together — reproducibly, in order.

docker-compose.yml
services:
  api:
    build: .
    ports:
      - "5000:5000"
    environment:
      - PORT=5000
      - MONGO_URI=mongodb://mongo:27017/mydb
    depends_on:
      - mongo

  mongo:
    image: mongo:7
    ports:
      - "27017:27017"
    volumes:
      - mongo-data:/data/db

volumes:
  mongo-data:
KeyWhat it doesWhy it matters
build: .Builds the api image from your local Dockerfile.No manual docker build step.
image: mongo:7Pulls Mongo straight from Docker Hub.No Dockerfile needed for off-the-shelf services.
depends_onStarts mongo before api.Controls startup order.
volumesDeclares the named volume for Mongo data.Data survives restarts (Section 09).
The magic: service names are hostnames

Notice the URI is mongodb://mongo:27017 — not localhost. Compose puts all services on one network where each is reachable by its service name. The api talks to mongo directly. No IP addresses, no host.docker.internal.

Start the whole stack
Terminal 
docker compose up -d          # build + start everything, detached
docker compose logs -f          # stream logs from all services
Tear it down
Terminal 
docker compose down            # stop & remove containers + network
docker compose down -v         # ...also delete volumes (wipes DB data!)
⚠️

docker compose down -v removes the volumes too — your Mongo data is permanently deleted. Plain down keeps data safe; add -v only when you truly want a clean slate.

Cheat Sheet
Print this. Pin it. The commands you'll actually reach for.

Images

Terminal 
docker build -t name .       # build image from Dockerfile

Containers

Terminal

docker stop api && docker rm api  # stop + remove

Env & Volumes

Terminal 
docker run --env-file .env name        # load env vars from file
docker run -v mongo-data:/data/db mongo:7  # persistent data

Compose

Terminal 
docker compose up -d      # start full stack
docker compose logs -f    # all service logs
docker compose down       # stop (keeps volumes)
docker compose down -v    # stop + wipe volumes

Cleanup

Terminal 
docker system prune -a    # reclaim disk (removes unused)
The one-sentence takeaway

Docker packages your code + environment into an image; you run that image as a container that behaves identically everywhere. Compose wires multiple containers (API + DB) into one stack. Volumes keep your data alive.
